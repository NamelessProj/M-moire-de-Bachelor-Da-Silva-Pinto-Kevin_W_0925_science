\section{Mise en \oe{}uvre} \label{chap:MediaProject}

Le Media Project constitue la partie centrale de ce travail. Il s'agit de concevoir et de développer \textbf{\ns}, un moteur de \gls{vn} \gls{opensource}, entièrement piloté par des fichiers texte au format \gls{json}. L'auteur rédige son histoire dans un fichier \gls{json} conforme au \gls{schema} défini, puis déploie sur le web, soit en compilant l'application \gls{reactjs} en fichiers statiques via \texttt{npm run build}, soit en utilisant une plateforme d'intégration continue telle que Vercel\footnote{Vercel est une plateforme de déploiement cloud spécialisée dans les applications web statiques et les frameworks \gls{js}. Elle automatise la compilation et la mise en ligne à chaque modification du dépôt source. \parencite{noauthor_vercel_nodate}} ou GitHub Pages\footnote{GitHub Pages est un service d'hébergement statique intégré à GitHub, permettant de publier automatiquement le contenu d'un dépôt sous forme de site web accessible via une URL publique, sans configuration serveur. \parencite{noauthor_documentation_nodate}}. Le \gls{vn} final est ainsi accessible depuis n'importe quel navigateur internet moderne, sans installation requise côté joueur. Le code source du projet est accessible publiquement sur GitHub à l'adresse suivante: \url{https://github.com/NamelessProj/NamelessStory}.

Le développement a suivi une approche itérative, chaque itération correspondant à un ensemble de fonctionnalités implémentées, testées et évaluées avant de passer à l'étape suivante.


\subsection{Évolution du périmètre: de la librairie à l'application} 

L'intention initiale de ce projet était de développer \nsits sous forme d'une \gls{librairie} \gls{reactjs} publiable sur \gls{npm}, intégrable par un développeur tiers via une simple commande d'installation. Cette approche aurait permis de distribuer le moteur comme un composant réutilisable, indépendamment de toute application hôte.

Afin d'apprendre à mettre en \oe{}uvre cette architecture, deux ressources ont été consultées: une playlist YouTube d'Imran Hussain (son nom d'utilisateur \textit{Imran Codes}) en 2024, intitulée \textit{Building a React Component Library: A Senior Developer's Guide}\parencite{imran_codes_1_2024} (ciblant \gls{reactjs} 16), et un article textuel d'Alex Eagleson publié sur dev.to en novembre 2021, mis à jour en décembre 2022 \parencite{noauthor_how_2021} (ciblant \gls{reactjs} 17 et rollup\footnote{Rollup est un outil de bundling \gls{js} spécialisé dans la création de \gls{librairie} distribuables. En mode \gls{librairie} (\textit{\gls{librairie} mode}), il compile le code source en un ou plusieurs fichiers optimisés, accompagnés des définitions de types \gls{ts}, prêts à être publiés comme \gls{package} \gls{npm}. \parencite{noauthor_introduction_nodate}} V2). Les deux ressources reposaient sur la même approche: configurer rollup en mode \gls{librairie} via un fichier \texttt{rollup.config.js}, en combinant les plugins \texttt{@rollup/plugin-typescript}, \texttt{@rollup/plugin-commonjs} et \texttt{rollup-plugin-dts}.

Dans les deux cas, la commande de build (\texttt{npm run build}) échouait. Les erreurs rencontrées relevaient de la configuration rollup, dont les exigences ont évolué avec les versions successives de l'outil. L'article d'Eagleson lui-même signale ce problème: une note intégrée au texte indique que certains outils ont été mis à jour et que les instructions telles qu'écrites ne fonctionnent plus. \parencite{noauthor_how_2021} Les commentaires des deux ressources, datant eux aussi d'environ un an, proposaient des correctifs partiels qui n'ont pas permis de résoudre le problème dans le contexte de \gls{reactjs} 19 et des versions actuelles de rollup.

Aucune ressource plus récente, reposant sur une configuration adaptée à \gls{reactjs} 19, n'a pu être identifiée, les tutoriels disponibles utilisaient tous la même approche de configuration, devenue incompatible avec les versions actuelles de l'outillage. Il n'est pas exclu que certaines erreurs rencontrées aient résulté d'une mauvaise interprétation des étapes de configuration de ma part, cette possibilité ne peut être écartée.

Face à ces obstacles et aux contraintes de temps du projet, la décision a été prise de réorienter l'architecture vers une application \gls{reactjs} autonome. Cette approche, plus directement maîtrisable, permet d'atteindre l'objectif principal du projet: rendre la création de \gls{vn} accessible via le web, sans nécessiter la distribution d'un \gls{package} \gls{npm}.

\subsubsubsection{Mécanisme d'optimisation React.js} \label{sec:optim_react}

\gls{reactjs} propose plusieurs mécanismes permettant de réduire le nombre de \glspl{rendu} inutiles et d'améliorer les performances d'une application. Trois d'entre eux ont été utilisés dans le cadre de ce projet \parencite{noauthor_reference_nodate}:

\texttt{React.memo} est une fonction qui \glslink{override}{court-circuite} le rendu d'un composant si ses \texttt{props} n'ont pas changé depuis le rendu précédent. Sans ce mécanisme, tout changement d'état d'un composant parent déclenche le recalcul de l'ensemble de ses composants enfants, qu'ils soient concernés ou non par le changement.

\texttt{useMemo} est un \gls{hook} \gls{reactjs} permettant de mettre en cache le résultat d'un calcul coûteux entre deux \glspl{rendu}. Le calcul n'est relancé que si l'une de ses dépendances déclarées a changé, évitant ainsi des recalculs inutiles à chaque \gls{rendu}.

\texttt{useRef} est un \gls{hook} permettant de persister une valeur entre les \glspl{rendu} sans déclencher de \glslink{rendu}{re-rendu} lors de sa modification. Il est utilisé pour stocker des références à des éléments du \acrshort{dom} ou des valeurs internes dont le changement ne doit pas provoquer de mise à jour de l'interface.


\subsubsection{Méthode de développement}

Le développement a suivi un processus itératif basé sur le cycle \textbf{PDCA} (\textit{Plan, Do, Check, Act}), tel que défini dans la méthodologie présentée au chapitre \ref{sec:PDCA}. Chaque itération correspond à un cycle complet: planification des fonctionnalités à implémenter, développement, évaluation par tests fonctionnels, puis décision sur les ajustements à apporter à l'itération suivante.

Ce modèle a été retenu pour sa capacité à intégrer les retours d'évaluation de manière continue, sans nécessiter de redéfinir l'ensemble du périmètre à chaque étape.


\subsection{Contraintes et limites}

\subsubsection{Contrainte de déploiement}

Contrairement aux moteurs de jeux traditionnels qui compilent le projet en un exécutable natif, \nsits repose sur des technologies web. Pour créer un \gls{vn} avec ce moteur, l'auteur doit télécharger le projet, puis rédiger son histoire dans un fichier \gls{json} selon le \gls{schema} défini. Le résultat peut ensuite être mis en ligne de deux manières:

\begin{itemize}
    \item En compilant l'application \gls{reactjs} en fichiers statiques via la commande \texttt{npm run build}, puis en hébergeant ces fichiers sur n'importe quel serveur web ou hébergeur statique.
    \item En connectant le dépôt à une plateforme d'intégration continue telle que Vercel ou GitHub Pages, qui se charge automatiquement de la compilation et du déploiement à chaque modification du projet.
\end{itemize}

\subsubsection{Contrainte de format}

Une contrainte fondamentale de ce projet est de rendre le contenu narratif lisible et interprétable à la fois par un humain et par la machine. le format \gls{json} a été retenu pour répondre à cette exigence: son modèle clé/valeur est suffisamment structuré pour être analysé de manière déterministe par le \gls{parseur}, tout en restant accessible à un auteur non-développeur disposé à respecter un \gls{schema} documenté.

Cependant, cette contrainte implique une limite inhérente: l'absence d'interprétation sémantique du langage naturel signifie que l'auteur doit respecter strictement la \gls{syntaxe} définie. Une erreur de structure dans le fichier \gls{json} peut bloquer l'exécution du moteur. La gestion de ces erreurs et le retour d'informations claires à l'auteur constituent donc un objectif de robustesse à adresser au fil des itérations.


\subsubsection{Contrainte de périmètre}

Pour des raisons de temps, le périmètre fonctionnel du moteur est limité aux mécaniques fondamentales des \gls{vn}, telles que définies dans le tableau \ref{tab:objectifs_moscow}. D'autres fonctionnalités, telles que la gestion de la musique de fond, des images de personnages ou d'un système de sauvegarde, sont identifiées comme souhaitables, mais ne font pas partie des objectifs primaires de ce travail.


\subsection{Étapes de développement}

Le développement de \nsits s'est déroulé en trois itérations successives, chacune suivant  le cycle PDCA défini au chapitre \ref{sec:PDCA}. Les objectifs de chaque itération ont été définis en amont selon la priorisation \gls{moscow} établie au tableau \ref{tab:objectifs_moscow}, et ajustés en fonction des résultats observés à l'issue de chaque cycle.


\subsubsection{Première itération: les bases d'un VN}


\subsubsubsection{Plan}

L'objectif de cette première itération était d'implémenter les mécaniques fondamentales du moteur: l'affichage progressif du texte (effet machine à écrire), la navigation entre scènes et la gestion des choix de dialogue. Ces trois fonctionnalités correspondent aux exigences \textit{Must} du tableau \ref{tab:objectifs_moscow}. Il s'agissait également de définir la structure du \gls{schema} \gls{json} servant de source de contenu, c'est-à-dire les types de n\oe{}uds que le moteur devrait être capable d'interpréter: dialogue, choix, scènes et personnages.


\subsubsubsection{Do}

J'ai commencé par créer le projet. Pour ce faire, il me fallait choisir un langage, un \gls{framework} et une structure. Pour le langage, j'ai opté pour \gls{ts}, afin de bénéficier d'un typage statique offrant une sécurité dès la phase de développement, avant même l'exécution. Concernant le \gls{framework}, j'ai hésité avec Next.js\footnote{Next.js est un \gls{framework} \gls{reactjs} orienté rendu serveur, développé par Vercel \parencite{noauthor_nextjs_nodate}}, mais ce dernier aurait été surdimensionné pour ce projet: \nsits ne nécessite aucun traitement côté serveur (\textit{\gls{backend}}), tout étant calculé dans le \gls{frontend}. J'ai donc opté pour \gls{reactjs}, une \gls{librairie} \gls{js} que je connais déjà et qui est parfaitement adaptée à la création de ce projet basé sur des composants, ce qui permet de séparer chaque partie du code.

Avant même de commencer à créer un seul composant, j'ai commencé à réfléchir à la structure que je souhaitais utiliser pour le fichier \gls{json} qui contiendra l'histoire. Je voulais quelque chose de personnalisable, afin de laisser un maximum de liberté aux personnes souhaitant utiliser ce projet. Pour atteindre cet objectif, il m'a fallu séparer plusieurs éléments:

Les personnages, à l'image des variables, sont susceptibles d'apparaître dans de nombreuses scènes. Les centraliser évite ainsi toute répétition dans le fichier.

Au fur et à mesure des itérations, le \gls{schema} s'enrichira afin de proposer davantage d'options. 

Une fois la structure définie, il me fallait commencer le développement. Pour ce faire, il fallait choisir par quoi commencer, et j'ai décidé de commencer par les composants visuels de base. C'est-à-dire par l'écran de titre, l'écran des crédits et un composant d'image de fond.

Comme la page de titre, les crédits et les scènes auront des images de fond. Pour rendre cet usage plus simple, j'ai pris la décision d'en faire un composant à part entière pour m'éviter des répétitions.

La raison pour laquelle j'ai commencé par les écrans de titre et de crédits au lieu du c\oe{}ur du projet, le \gls{vn}, est que je voulais d'abord me consacrer aux éléments simples qui n'allaient pas avoir beaucoup d'évolution. Une fois ces écrans réalisés, je pouvais me consacrer entièrement au \gls{vn} tout en sachant que les autres écrans étaient déjà faits.

Une fois ces éléments réalisés, j'ai commencé à travailler sur le composant que je considère comme le plus important: l'affichage du texte, avec l'effet machine à écrire. Pour rendre ce composant plus puissant, mon intention était de faire en sorte qu'il puisse \glslink{parseur}{parser} des balises \acrshort{html} et de pouvoir mettre en place une \gls{syntaxe} de pauses afin de donner du tonus aux textes. J'ai donc créé un système de \gls{tokenisation} du texte. L'idée est simple: avant l'affichage, le texte brut est découpé en unités atomiques, les \textit{tokens}, qui peuvent être soit du texte ordinaire, soit des balises \acrshort{html}, ou encore des temps de pause, comme l'illustre la figure \ref{fig:tokenisation-flux}. Cette séparation permet à l'effet machine à écrire de n'animer que les caractères visibles, sans traiter les balises comme du texte à afficher. Sans cette étape intermédiaire, afficher \texttt{<b>mot</b>} caractère par caractère aurait risqué de produire des balises partielles visibles à l'écran, ou un \acrshort{html} mal formé.

\subsubsection{Deuxième itération: l'optimisation}

\subsubsubsection{Plan}

L'objectif de cette itération était d'améliorer les performances et la maintenabilité du code produit lors de la première itération. Dans \gls{reactjs}, chaque mise à jour de l'état d'un composant parent déclenche un nouveau rendu de l'ensemble de l'arbre de composants enfants. Dans un moteur de \gls{vn}, où l'état global évolue à chaque ligne de dialogue, ce comportement peut provoquer des recalculs inutiles sur des composants dont les données n'ont pas changé. un second axe concernait la gestion de la sauvegarde: l'opération de sérialisation de l'état courant via \texttt{JSON.stringify()} s'exécutait sur le fil principal du navigateur. Cette opération ne pose pas de problème tant que l'objet à sérialiser reste de taille modeste. Cependant, à mesure que l'état s'enrichit (variables de jeu, historique de dialogue, données de progression), cette opération peut devenir coûteuse et bloquer l'interface le temps de son exécution. Afin d'anticiper cette limitation, la décision a été prise de déléguer cette opération à un \gls{webworker}.


\subsubsubsection{Do}

L'optimisation du rendu a porté sur l'ensemble des composants du moteur. Chaque composant a été mémorisé via \texttt{React.memo} (voir section \ref{sec:optim_react}), ce qui permet à \gls{reactjs} de court-circuiter le rendu d'un composant si ses \textit{props} n'ont pas changé depuis le rendu précédent. Pour les calculs coûteux effectués à l'intérieur des composants, le \textit{hook} \texttt{useMemo} a été utilisé afin de mettre les résultats en cache entre les rendus. Des références (\texttt{useRef}) ont également remplacé certains états React où une valeur devait persister sans déclencher de re-rendu. Le composant \texttt{Typewriter} a notamment fait l'objet d'une refactorisation significative: les instantanés \acrshort{html} et les délais entre caractères sont désormais précalculés en une seule passe avant le début de l'animation, évitant tout recalcul durant la frappe.

L'extrait de code \ref{lst:typewriter} illustre la stratégie d'optimisation du composant \texttt{Typewriter}. Avant le début de l'animation, \texttt{precomputeSteps} calcule en une seule passe l'ensemble des \glslink{snaphtml}{snapshots HTML} et des délais associés à chaque caractère. Ces snapshots sont ensuite convertis en n\oe{}uds React via un second \texttt{useMemo}, dont les dépendances sont strictement séparées du premier. Durant l'animation, le \texttt{useEffect} de tick se limite à lire \texttt{nodeSnapshots[currentStep]} et de planifier le prochain appel via \texttt{setTimeout} avec le délai précompute \texttt{delays[currentStep]}: aucun calcul n'est effectué à chaque frappe. 

\begin{listing}[H]
\begin{minted}{ts}
// Pré-calcul unique : 0(n) avant l'animation
const precomputed = useMemo(
    () => TypewriterUtils.precomputeSteps(tokens, speed),
    [tokens, speed]
);

// Conversion des snapshots HTML en nœuds react (une seule fois)
const nodeSnapshots: ReactNode[] = useMemo(
    () => precomputed.htmlSnapshots.map(htmlToReactNode),
    [precomputed]
);

// Avancement : lecture directe à l'index | 0(1) par tick
useEffect(() => {
    if (currentStep >= totalSteps) {
        if (!completedRef.current) {
            completedRef.current = true;
            onComplete?.();
        }
        return;
    }
    const delay: number = precomputed.delays[currentStep] ?? speed;
    const timeout: number = setTimeout(
        () => setCurrentStep((prev) => prev + 1),
        delay
    );
    return () => clearTimeout(timeout);
}, [currentStep, totalSteps, precomputed, speed, onComplete]);
\end{minted}
\caption{Précomputation des instantanés HTML et des délais dans le composant \textit{Typewriter} (Auteur 2026)}
\label{lst:typewriter}
\end{listing}

En parallèle, le traitement de la mise en forme du texte a été revu pour des raisons de sécurité et de robustesse. La propriété React \texttt{dangerouslySetInnerHTML} qui permettait d'injecter directement du \acrshort{html} dans le \acrshort{dom}, a été remplacée par un couple de fonctions dédiées: \texttt{escapeHtml}, qui neutralise les caractères spéciaux avant tout traitement, et \texttt{htmlToReactNode}, qui convertit la chaîne résultante en n\oe{}uds React natifs. Cette approche supprime le risque d'injection de contenu malveillant tout en conservant la flexibilité du balisage personnalisé. Pour garder cette personnalisation des balises dans le texte, un \gls{parseur} inspiré de \gls{markdown} a été mis en place.

Pour la sauvegarde, un \gls{webworker} dédié (\texttt{saveWorker}) a été mis en place. Le fil principal transmet l'état courant au \textit{worker} par échange de messages, celui-ci exécute \texttt{JSON.stringify()} dans son propre fil d'exécution, puis retourne la chaîne sérialisée au fil principal, comme illustré à la figure \ref{fig:webworker-architecture}. Cette communication est entièrement asynchrone: l'interface utilisateur reste réactive pendant toute la durée de l'opération, quelle que soit la taille de l'objet à sérialiser.

L'extrait de code \ref{lst:saveworker} présente les deux composants de cette architecture. Côté fil secondaire, le \texttt{worker} reçoit via \texttt{postMessage} un objet contenant l'état courant et le nom du cookie cible, exécute \texttt{JSON.stringify} dans son propre fil d'exécution, puis retourne la chaîne sérialisée accompagnée du nom du cookie. Ce couplage garantit que le \textit{worker} ne conserve aucune donnée entre deux appels: il reçoit l'intégralité des informations nécessaires à chaque message et peut être réutilisé pour n'importe quelle opération de sauvegarde sans initialisation supplémentaire. Côté fil principal, le \textit{worker} est instancié une seule fois à l'initialisation du composant via \texttt{useEffect} et est conservé dans une référence \texttt{useRef} pour éviter des instanciations répétées. La fonction de rappel \texttt{onmessage} reçoit le résultat de manière asynchrone et déclenche l'écriture du cookie sans jamais avoir bloqué le fil de rendu.

// workers/saveWorkers.ts (fil secondaire)
self.addEventListener("message",
    (e: MessageEvent<SaveWorkerMessage>) => {
        const { state, cookieName } = e.data;
        const serialized = JSON.stringify(state);
        (self as unknown as Worker).postMessage(
            { serialized, cookieName } satisfies SaveWorkerResponse
        );
    }
);


// VisualNovel/index.tsx (fil principal)
const saveWorkerRef = useRef<Worker | null>(null);

useEffect(() => {
    const worker = new Worker(
        new URL('../../../workers/saveWorker.ts', import.meta.url),
        { type: 'module' }
    );
    worker.addEventListener('message',
        (e: MessageEvent<SaveWorkerResponse>) => {
            saveState(e.data.cookieName, e.data.serialized);
        }
    );
    saveWorkerRef.current = worker;
    return () => {
        worker.terminate();
        saveWorkerRef.current = null;
    };
}, []);
\end{minted}

\subsubsection{Troisième itération: la sécurité et la robustesse}

\subsubsubsection{Plan}

L'objectif de cette itération était d'adresser la limite principale identifiée lors des itérations précédentes: l'absence de validation formelle des fichiers \acrshort{json} fournis par l'auteur. Sans mécanisme de vérification, un fichier syntaxiquement  invalide ou non conforme au \gls{schema} attendu pouvait mettre le moteur dans un état indéterminé, sans retour d'information exploitable pour l'auteur. L'intégration de la \gls{librairie} \gls{zod}, présentée au chapitre \ref{sec:zod}, constituait la réponse à cette lacune.

Par ailleurs, la découverte de \gls{zod} n'a pas été le fruit d'une recherche ciblée, mais d'une rencontre fortuite sur GitHub, où la \gls{librairie} apparaissait fréquemment dans des projets \gls{ts} traitant de la validation de \gls{schema}. Avant de l'adopter, deux alternatives ont été évaluées: TypeBox\footnote{\textbf{TypeBox}: \gls{librairie} de validation de \gls{schema} \gls{ts} basé sur \gls{json}, développée par Haydn Paterson \parencite{noauthor_typebox_nodate}}  et ArkType\footnote{\textbf{ArkType}: \gls{librairie} de validation et d'inférence de types pour \gls{ts}, axée sur la performance et une \gls{syntaxe} inspirée de la notion native du langage \parencite{noauthor_arktype_nodate}}. TypeBox propose une approche similaire à \gls{zod}, tandis qu'ArkType repose sur une \gls{syntaxe} sensiblement différente, dont la prise en main s'est révélée moins immédiate. Le choix s'est finalement porté sur \gls{zod} pour la concision et la lisibilité de sa \gls{syntaxe}, un critère important dans l'optique de maintenir un code accessible aux contributeurs potentiels du projet.

Par ailleurs, cette itération a permis d'implémenter plusieurs fonctionnalités \textit{Can} identifiées au tableau \ref{tab:objectifs_moscow}, dont le temps disponible n'avait pas permis l'intégration lors des itérations précédentes: l'affichage des \textit{sprites} de personnages avec positionnement dynamique, les transitions entre scènes et dialogues, la gestion des polices d'écriture via des variables \acrshort{css}, et la prise en charge d'un \textit{placeholder} dynamique pour les champs de saisie.


\subsubsubsection{Do}

L'intégration de \gls{zod} s'est articulée autour de deux \glspl{schema} de validation distincts. Le premier, \texttt{storySchema}, décrit la structure attendue du fichier \acrshort{json} de scénario: types de n\oe{}uds, présence des champs obligatoires, valeurs autorisées pour les propriétés énumérées. Le second, \texttt{saveSchema}, couvre la structure des fichiers de sauvegarde afin de détecter et de rejeter tout fichier corrompu ou modifié manuellement avant qu'il ne soit transmis au moteur. En cas d'écart par rapport au \gls{schema} défini (champ manquant, type incorrect, structure invalide), \gls{zod} retourne un message d'erreur explicite indiquant précisément la localisation et la nature du problème, ce qui constitue un mécanisme de détection d'erreurs déterministe et un retour direct pour l'auteur.

L'extrait de code \ref{lst:storyschema} présente un extrait du \texttt{VNStorySchema}. La validation s'articule en deux niveaux. le premier est déclaratif: chaque champ du fichier \gls{json} est typé explicitement via les primitives \gls{zod} (\texttt{z.string()}, \texttt{z.number().positive()}, \texttt{z.enum([...])}), ce qui permet de détecter à l'exécution tout champ manquant, tout type incorrect ou toute valeur hors des options autorisées. Le second niveau, implémenté via \texttt{superRefine}, effectue une validation sémantique cross-champs impossible à exprimer avec des types seuls: il vérifie que la scène référencée par \texttt{startingScene} existe bien dans le bloc \texttt{story} et que l'ensemble des cibles de navigation (champs \texttt{next} des dialogues et des options) pointe vers des scènes déclarées. En cas d'écart, \texttt{ctx.addIssue} produit un message d'erreur incluant le chemin exact du champ fautif, tel qu'illustré à la figure \ref{fig:zod_erreurs}.

\begin{listing}[H]
\begin{minted}{ts}
const DialogueSchema = z.object({
    name: z.string(),
    text: z.string(),
    textSpeed: z.number().positive().optional(),
    options: z.array(OptionSchema).optional(),
    sprite: SpriteSchema.optional(),
    next: z.string().optional(),
    transition: DialogueTransitionSchema.optional(),
    // ...
});

export const VNStorySchema = z.object({
    settings: SettingsSchema,
    characters: z.record(z.string(), CharacterTypeSchema),
    story: z.record(z.string(), SceneTypeSchema),
}).superRefine((data, ctx) => {
    // Vérifie que startingScene référence une scène existante
    if (!data.story[data.settings.startingScene]) {
        ctx.addIssue({
            code: "custom",
            message: `"${data.settings.startingScene}" does not exist in Story`,
            path: ["settings", "startingScene"],
        });
    }
    // Vérifie que toutes les cibles de navigation existent
    for (const [sceneId, scene] of Object.entries(data.story)) {
        scene.dialogues.forEach((dialogue, i) => {
            const targets = [
                ...(dialogue.options ?? []).map(
                    o => ({value: o.next, path: [/*...*/, i, "options"]})
                ),
                ... (dialogue.next ? [
                    {value: dialogue.next, path: [/*...*/, i, "next"]}
                ] : []),
            ];
            for (const target of targets) {
                const [targetScene] = target.value.split(":");
                if (!data.story[targetScene] && targetScene !== END_STORY_TOKEN) {
                    ctx.addIssue({
                        code: "custom",
                        message: `Scene "${targetScene}" does not exist`,
                        path: ["story", ...target.path],
                    });
                }
            }
        });
    }
});
\end{minted}
\caption{Extrait du \texttt{VNStorySchema}: validation déclarative et vérification sémantique des références de navigation (Auteur 2026)}
\label{lst:storyschema}
\end{listing}

Afin de couvrir les cas d'erreur de manière systématique, des fichiers de test dédiés ont été ajoutés au dépôt dans un dossier \texttt{tests/}. Ces fichiers représentent intentionnellement des scénarios invalides: \acrshort{json} à la \gls{syntaxe} cassé (virgule manquante, guillemet non fermé), fichiers avec des champs obligatoires absents, et fichiers avec des valeurs de type incorrect.
